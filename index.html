üì¶ Step 1: Understanding Pods - The Foundation

Welcome! Let's start with Pods - the smallest deployable unit in Kubernetes. A Pod wraps around one or more containers and provides shared resources.

What's a Pod?

A Pod is like a wrapper that contains your application container. It provides:

  • üåê Shared networking - All containers share the same IP
  • üíæ Shared storage - Volumes are shared between containers
  • üìã Shared configuration - Environment and specifications
apiVersion: v1 kind: Pod metadata: name: nginx-pod spec: containers: - name: nginx image: nginx:1.14.2 ports: - containerPort: 80
üéØ Key Point: Think of a Pod as a "house" where your app lives. The containers inside share everything - like roommates sharing the same address, internet connection, and storage space.

üõ°Ô∏è Step 4: Deployments - Auto-Healing & Scaling

A Deployment manages ReplicaSets, which manage Pods. It provides auto-healing, scaling, and rolling updates.

apiVersion: apps/v1 kind: Deployment metadata: name: nginx-deployment spec: replicas: 3 selector: matchLabels: app: nginx template: metadata: labels: app: nginx spec: containers: - name: nginx image: nginx:1.14.2 ports: - containerPort: 80
Creating Deployment
kubectl@cluster:~$

Let's see what the Deployment created:

üéâ Perfect! The Deployment created:
  • 1 ReplicaSet (manages pod replicas)
  • 3 Pods (as specified by replicas: 3)
  • All with cluster IPs and kube-proxy load balancing

üîÑ Step 5: Test Auto-Healing Magic

Now let's see the real magic - when a Pod crashes, the ReplicaSet automatically creates a new one!

Simulating Pod Crash
kubectl@cluster:~$
kubectl delete pod nginx-deployment-xyz-abc-1
pod "nginx-deployment-xyz-abc-1" deleted

Watch the magic happen:

kubectl@cluster:~$
kubectl get pods -w
NAME READY STATUS RESTARTS AGE nginx-deployment-xyz-abc-2 1/1 Running 0 2m nginx-deployment-xyz-abc-3 1/1 Running 0 2m nginx-deployment-xyz-abc-4 1/1 Running 0 5s
üîÆ Auto-Healing in Action! The ReplicaSet noticed we only had 2 pods but needed 3, so it immediately created a new pod (nginx-deployment-xyz-abc-4). Zero downtime!
üß† This is ReplicaSet magic:
  • Desired state: 3 pods
  • Actual state: 2 pods (after deletion)
  • Action: Create 1 new pod immediately
  • Result: Back to 3 pods running

üîÑ Step 7: ReplicaSets - The Controller Behind the Magic

Let's understand what a ReplicaSet actually does. It's the controller that ensures the desired number of pods are always running.

üß† ReplicaSet Logic:
  1. Check current number of pods
  2. Compare with desired replicas
  3. If less ‚Üí Create new pods
  4. If more ‚Üí Delete excess pods
  5. Repeat every few seconds
Inspecting ReplicaSet Details
kubectl@cluster:~$
kubectl describe replicaset nginx-deployment-xyz-abc
Name: nginx-deployment-xyz-abc Namespace: default Selector: app=nginx,pod-template-hash=xyz-abc Labels: app=nginx Annotations: deployment.kubernetes.io/desired-replicas: 2 Replicas: 2 current / 2 desired Pods Status: 2 Running / 0 Waiting / 0 Succeeded / 0 Failed Pod Template: Labels: app=nginx Containers: nginx: Image: nginx:1.14.2 Port: 80/TCP Events: Type Reason Age From Message ---- ------ ---- ---- ------- Normal SuccessfulCreate 7m replicaset-controller Created pod: nginx-deployment-xyz-abc-2 Normal SuccessfulCreate 7m replicaset-controller Created pod: nginx-deployment-xyz-abc-3

Notice the key information:

  • Selector: How it finds pods to manage (app=nginx)
  • Replicas: 2 current / 2 desired (perfect match!)
  • Events: Shows the creation history
‚ö†Ô∏è Important: ReplicaSets handle scaling (replicas count) but deployments handle updates. When you update your image, the Deployment creates a NEW ReplicaSet and gradually replaces the old one.

üéØ Interview Questions - You're Ready!

Q: What's the difference between a Pod, ReplicaSet, and Deployment?

Pod: The smallest unit - wraps containers, no auto-healing
ReplicaSet: Manages multiple identical pods, ensures desired count
Deployment: Manages ReplicaSets, handles updates and rollbacks
Q: Why use Deployments instead of Pods?

Deployments provide:
‚Ä¢ Auto-healing (replace crashed pods)
‚Ä¢ Auto-scaling (adjust to load)
‚Ä¢ Rolling updates (zero-downtime deployments)
‚Ä¢ Rollback capabilities
‚Ä¢ Production-ready management
